## Purpose

These documents collectively define the technical architecture and operational behavior of the InFlux protocol.

Each specification is intended to be deterministic, implementation-agnostic, and suitable for long-term maintenance.

---

## Document Relationships

The specifications are designed to be read in the following order:

1. System Architecture
2. Consensus Engine
3. Economic Engine
4. Validator Lifecycle
5. State Replication
6. Cluster Formation
7. Cross-Cluster Synchronization
8. Economic Propagation
9. Tokenomics
10. Governance
11. Network Security
12. Mainnet Deployment

Later documents build upon concepts introduced earlier in the sequence.

---

## Maintenance

Protocol specifications should be updated whenever a protocol change affects architecture, networking, consensus, economics, governance, or deployment.
